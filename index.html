Chapter 01

유저 테스트란
무엇인가

유저 테스트(UT, Usability Testing)는 실제 사용자가 제품(앱, 웹사이트, 서비스)을 사용하는 모습을 관찰하면서 문제점을 발견하는 연구 방법입니다. 디자이너나 기획자의 머릿속 가정이 아니라, 실제 행동 데이터를 기반으로 제품을 개선합니다.

요리사가 새 메뉴를 만들었다고 상상해 보세요. 주방에서 혼자 먹어보고 "맛있다" 판단하는 것과, 손님 10명에게 직접 먹여보고 반응을 관찰하는 것 — 어느 쪽이 더 정확할까요? 유저 테스트는 후자입니다. 내가 만든 것이 아니라, 쓰는 사람의 눈으로 보는 것입니다.

왜 필요한가: "나는 사용자가 아니다"

제품을 만드는 사람은 그 제품을 너무 잘 알기 때문에, 실제 사용자가 어디서 헤매는지 보이지 않습니다. 이를 '전문가의 저주(Curse of Knowledge)'라고 합니다. UT는 이 인식의 간격을 메우는 도구입니다.

🔍

발견

우리가 미처 몰랐던 사용성 문제를 발굴합니다. 예: "회원가입 버튼이 안 보인다"

✅

검증

새로운 디자인 방향이 실제로 효과가 있는지 확인합니다. 가설이 맞는지 테스트합니다.

📊

근거

팀 내 의사결정에 "내 생각에는…" 대신 "데이터에 따르면…"을 쓸 수 있게 됩니다.

💸

비용 절감

개발 전에 문제를 발견할수록 수정 비용이 줄어듭니다. 늦을수록 비쌉니다.

Chapter 02

유저 테스트
방법론

UT에는 여러 방법이 있습니다. 상황과 목적에 따라 다른 방법을 씁니다. 크게 정성적(Qualitative) 방법과 정량적(Quantitative) 방법으로 나뉩니다.

📌

정성 vs 정량

정성적: "왜 그랬는지" 이유와 맥락을 이해합니다. 인터뷰, 관찰처럼 사람의 말과 행동을 분석합니다. (깊이 있는 이해)
정량적: "몇 명이 그랬는지" 숫자로 파악합니다. 설문, 클릭 데이터처럼 측정 가능한 데이터를 분석합니다. (규모 파악)

A

심층 인터뷰 (In-depth Interview)

1:1로 사용자를 만나 30~60분 대화합니다. 제품 사용 경험, 불편한 점, 필요한 것들을 자유롭게 이야기합니다. 수치화는 어렵지만 숨겨진 니즈(Latent Need)를 발견하는 데 최적입니다.

B

사용성 테스트 (Usability Testing)

사용자에게 실제 태스크를 주고 수행하는 과정을 관찰합니다. "장바구니에 상품을 담고 결제해 주세요"처럼 구체적인 미션을 주고, 어디서 막히는지, 어디서 헤매는지를 기록합니다. 싱크 어라우드(Think Aloud) 기법—생각나는 것을 소리내어 말하게 하기—을 함께 씁니다.

C

컨텍스추얼 인콰이어리 (Contextual Inquiry)

사용자의 실제 환경(사무실, 집, 카페 등)에 직접 찾아가 관찰합니다. 실험실 환경이 아닌 실제 맥락(Context)에서 행동을 보기 때문에 더 자연스러운 데이터를 얻습니다.

D

A/B 테스트

두 가지 버전(A안과 B안)을 실제 사용자에게 동시에 노출시켜 어느 쪽이 더 효과적인지 수치로 비교합니다. 버튼 색깔, 문구, 레이아웃 등 변수를 하나씩 바꿔 테스트합니다. 대규모 트래픽이 있는 서비스에서 특히 유효합니다.

E

설문조사 (Survey)

많은 사용자에게 동일한 질문을 던져 패턴을 수치로 파악합니다. 빠르고 대규모지만 깊이가 얕습니다. NPS(Net Promoter Score, 추천 의향 점수) 측정 등에 활용됩니다.

F

히트맵 & 세션 레코딩 (Heatmap & Session Recording)

Hotjar, FullStory 같은 툴로 사용자가 화면 어디를 클릭하고, 어디까지 스크롤하는지 시각화합니다. 사용자가 실제로 어디에 주의를 기울이는지 행동 데이터를 자동으로 수집합니다.

Chapter 03

행동 매핑
(Behavior Mapping)

행동 매핑은 사용자가 특정 목표를 달성하기 위해 취하는 모든 행동을 순서대로 시각화하는 방법입니다. "사용자가 실제로 무엇을 하는가"를 지도처럼 펼쳐놓습니다.

네비게이션 앱을 생각해 보세요. 앱을 켜고 목적지를 검색하고 경로를 선택하고 안내를 따라 이동하는 과정 — 이 각각의 단계에서 사람들이 실제로 어떻게 행동하는지, 어디서 막히는지, 무엇을 기대하는지를 지도처럼 그리는 것이 행동 매핑입니다.

행동 매핑 vs 유저 저니 맵

비슷해 보이지만 관점이 다릅니다. 유저 저니 맵(User Journey Map)은 사용자의 감정과 경험 전체를 큰 그림으로 그리는 것이고, 행동 매핑은 특정 기능이나 플로우 안에서의 구체적인 행동 단계에 집중합니다.

// BEHAVIOR MAP EXAMPLE — 음식 배달 앱 주문 플로우
단계 사용자 행동 기대 (사용자) 실제 발생 문제 감정 상태 기회 영역
01 진입 앱 실행 → 메인 화면 스캔 빠르게 원하는 음식 찾기 광고 배너가 상단 점령, 검색이 안 보임 😐 중립 검색창 prominence 개선
02 탐색 카테고리 탭 이동 → 스크롤 빠른 필터링, 가격 확인 필터가 너무 많아 어디서 시작해야 할지 모름 😕 혼란 기본 필터 단순화
03 선택 음식점 클릭 → 메뉴 보기 메뉴 사진 + 가격 한눈에 사진 없는 메뉴 多, 구성품 불명확 😟 불안 사진 업로드 가이드 강화
04 담기 수량 선택 → 장바구니 추가 선택이 잘 반영됐는지 확인 담기 후 확인 피드백 없음, 다시 메뉴로 돌아가야 함 😤 짜증 담기 성공 토스트 + 연속 담기 UX
05 결제 장바구니 → 결제 화면 총 금액 + 배달비 바로 확인 배달비가 결제 직전에 나타남, 이탈 급증 😡 배신감 🔥 배달비 조기 노출 (최우선)
06 완료 주문 완료 → 배달 추적 예상 시간, 실시간 위치 배달 현황 업데이트 느림 😐 대기 배달원 실시간 위치 표시

행동 매핑을 만드는 방법

1
범위 설정
어떤 플로우를 분석할 것인가 결정
2
데이터 수집
UT 영상, 세션 레코딩, 인터뷰
3
행동 분해
각 단계별 행동을 최소 단위로 쪼갬
4
문제 마킹
각 단계에서 실제 발생한 문제 기록
5
패턴 찾기
여러 사용자에서 반복되는 문제 발굴
⚡

핵심: 행동과 해석을 분리하라

행동 매핑에서 가장 중요한 원칙은 "사용자가 한 행동"과 "그 행동에 대한 나의 해석"을 분리하는 것입니다. 사용자가 결제 버튼을 3번 눌렀다는 것은 사실(fact)이고, "버튼이 헷갈렸다"는 것은 해석(interpretation)입니다. 먼저 사실을 정확히 기록해야 합니다.

Chapter 04

데이터 가공:
Raw Data → Insight

테스트를 마쳤습니다. 이제 수십 페이지의 메모, 영상 녹화, 설문 결과가 손에 있습니다. 이것을 어떻게 실행 가능한 인사이트로 바꿀까요? 업계에서 쓰는 주요 방법을 단계별로 설명합니다.

01
RAW DATA

원시 데이터 정리

테스트 직후, 기억이 선명할 때 모든 관찰을 기록합니다.

  • 테스트 영상 타임스탬프 기록 ("2:34 — 결제 버튼 3번 클릭, 혼란스러워 함")
  • 관찰자 노트 디지털화
  • 설문 데이터 취합 (스프레드시트)
  • 인터뷰 녹취 텍스트 변환
02
CODING

코딩 (Qualitative Coding)

각 관찰 내용에 의미 있는 레이블(코드)을 붙입니다. 정성 데이터를 분류 가능하게 만드는 작업입니다.

  • "결제 버튼 못 찾음" → 코드: #navigation_failure
  • "배달비 보고 놀람" → 코드: #price_transparency
  • "사진 없어서 주문 망설임" → 코드: #information_insufficient
03
CLUSTERING

어피니티 다이어그램 (Affinity Diagram)

비슷한 관찰들을 묶어 주제 클러스터를 만듭니다. 포스트잇을 벽에 붙이고 비슷한 것끼리 모으는 방식입니다 (KJ 방법이라고도 함). 개별 관찰들이 더 큰 패턴으로 보이기 시작합니다.

  • 팀 전체가 함께 수행해야 편향을 줄일 수 있음
  • 디지털 도구: FigJam, Miro, Notion
04
ANALYSIS

패턴 분석 및 빈도 측정

어떤 문제가 몇 명의 사용자에게 발생했는지 세고, 심각도(Severity)를 평가합니다.

  • 빈도(Frequency): 5명 중 4명이 겪은 문제 vs 1명만 겪은 문제
  • 심각도: 태스크 완전 실패 / 지연 / 불편 3단계로 분류
  • 태스크 완료율(Task Completion Rate) 계산
  • 오류율(Error Rate), 소요 시간(Time on Task) 측정
05
INSIGHT

인사이트 도출

분석 결과를 행동 가능한 문장으로 만듭니다. 단순 사실 나열이 아닌, 의미를 담은 진술입니다.

  • 관찰: "5명 중 4명이 결제 직전 배달비를 보고 뒤로 갔다"
  • 인사이트: "배달비 정보가 너무 늦게 노출되어 사용자의 가격 기대치와 충돌, 이탈을 유발한다"
  • 기회: "메뉴 탐색 단계부터 배달비 예상 금액을 표시한다"
Detail View

어피니티 다이어그램
실제 예시

배달 앱 UT에서 수집한 20개의 관찰을 4개의 클러스터로 묶은 예시입니다.

🔴 정보 투명성 (6건)
배달비가 마지막에 나온다, 놀랐다
메뉴에 뭐가 들어가는지 몰라서 못 시켰다
사진이 없어서 상상으로 주문했다
최소주문금액을 못 봤다가 결제 직전에 알았다
🔵 내비게이션 (5건)
검색 어디 있는지 한참 찾았다
카테고리가 너무 많아서 어디서 시작할지 몰랐다
장바구니로 가는 버튼이 어디 있는지 헷갈렸다
메뉴 보다가 어떻게 돌아가는지 몰랐다
🟡 피드백 부재 (5건)
담았는데 담긴 건지 몰랐다, 두 번 눌렀다
주문이 됐는지 확인이 안 됐다
결제 중인지 오류인지 알 수가 없었다
로딩 화면이 너무 길어서 꺼진 줄 알았다
🟢 신뢰 & 리뷰 (4건)
리뷰가 오래된 것들이라 믿기 어려웠다
별점이 다 비슷해서 기준이 없었다
사장님 댓글이 없는 곳은 불안했다
단골 표시 같은 게 있으면 좋겠다
Chapter 05

인사이트를
디자인으로

인사이트가 나왔다면, 이제 그것을 실제 디자인 개선으로 연결해야 합니다. 업계에서 자주 쓰는 방법이 HMW(How Might We) 질문법입니다.

HMW (How Might We) 질문법

인사이트를 "어떻게 하면 우리가 ~할 수 있을까?" 형식의 질문으로 바꿉니다. "문제가 있다"는 사실 진술에서 "해결을 탐색"하는 질문으로 전환하는 것입니다. 이 질문이 아이디에이션(Ideation)의 시작점이 됩니다.

HMW

사용자가 메뉴를 탐색하는 시작 단계부터 배달비를 자연스럽게 확인할 수 있도록 할 수 있을까?

HMW

메뉴를 장바구니에 담았다는 명확한 확신을 사용자에게 즉각적으로 줄 수 있을까?

HMW

수십 가지 필터 대신, 사용자가 원하는 것을 3번의 탭 이내에 찾도록 도울 수 있을까?

HMW

오래된 리뷰와 신뢰도 낮은 별점 시스템을 개선해 더 믿을 수 있는 판단 기준을 제공할 수 있을까?

인사이트는 발견이고,
HMW는 질문이고,
디자인은 답이다.

인사이트 → 디자인 변경 연결 구조

🔬 관찰 (What)

테스트 참여자 6명 중 5명이 결제 화면에서 뒤로 가기를 눌렀다. 3명은 "배달비가 이렇게 비쌀 줄 몰랐다"고 발언했다.

💡 인사이트 (Why)

배달비 정보가 결제 최종 단계에서만 노출되어, 메뉴 선택 과정에서 형성된 가격 기대치와 충돌이 발생한다. 이 충돌이 이탈을 유발한다.

🎯 디자인 액션 (How)

음식점 목록 카드에 예상 배달비를 표시한다. 장바구니 상단에 실시간 배달비를 고정 표시한다. 총 예상 금액을 항상 보여준다.

Chapter 06

우선순위 결정:
무엇을 먼저 고칠까

문제가 10개 발견됐다고 10개를 동시에 고칠 수는 없습니다. 리소스(시간, 인력)는 한정되어 있습니다. 우선순위 매트릭스(Priority Matrix)를 활용해 무엇부터 할지 결정합니다.

병원 응급실에 비유하면, 경미한 상처와 심장마비 환자가 동시에 왔을 때 중증도에 따라 순서를 정하는 것과 같습니다. 디자인에서도 사용자에게 미치는 영향이 크고, 구현이 쉬운 것을 먼저 합니다.

2×2 우선순위 매트릭스

사용자 임팩트(Impact)와 구현 난이도(Effort)를 두 축으로 놓고, 발견된 문제들을 배치합니다.

Quick Wins를 먼저

높은 임팩트와 낮은 구현 난이도를 가진 항목을 먼저 실행합니다. 빠른 성과를 내면서 팀의 신뢰를 쌓고, 리서치 기반 의사결정이 실제로 효과가 있다는 것을 증명합니다. 작은 승리들이 더 큰 프로젝트를 위한 동력이 됩니다.

반복(Iteration): 고쳤으면 다시 테스트

개선 후에는 반드시 다시 테스트합니다. 고친 것이 실제로 문제를 해결했는지, 혹은 새로운 문제를 만들어내지는 않았는지 확인합니다. 이 사이클이 반복되면서 제품이 점점 좋아집니다. 이것이 반복적 설계(Iterative Design)의 핵심입니다.

핵심 정리

// KEY CONCEPTS

✓
유저 테스트는 내 추측이 아닌 사용자의 실제 행동을 근거로 제품을 개선하는 방법이다
✓
정성적 방법(인터뷰, 관찰)은 왜를 이해하고, 정량적 방법(A/B 테스트, 설문)은 얼마나를 측정한다
✓
행동 매핑은 사용자의 행동 단계를 지도처럼 펼쳐놓고, 어디서 문제가 발생하는지를 시각화한다
✓
어피니티 다이어그램으로 개별 관찰들을 클러스터링해 패턴(인사이트)을 도출한다
✓
인사이트는 HMW 질문으로 변환해 구체적인 디자인 방향으로 연결한다
✓
우선순위 매트릭스로 임팩트와 난이도를 기준으로 무엇을 먼저 고칠지 결정한다
✓
개선 → 테스트 → 개선의 반복(Iteration)이 좋은 제품을 만드는 과정이다

업계 핵심 용어 레퍼런스

Usability Testing

실제 사용자가 태스크를 수행하는 것을 관찰하는 테스트

Think Aloud

테스트 중 생각을 소리내어 말하게 하는 기법

Affinity Diagram

관찰 내용을 비슷한 것끼리 묶어 패턴을 찾는 방법

HMW (How Might We)

인사이트를 디자인 기회로 변환하는 질문 프레임

Behavior Mapping

사용자의 행동 단계를 순서대로 시각화하는 방법

Iterative Design

테스트-개선을 반복하며 점진적으로 완성도를 높이는 프로세스

Task Completion Rate

주어진 태스크를 완료한 사용자의 비율

Qualitative Coding

관찰 내용에 레이블을 붙여 분류 가능하게 만드는 작업
